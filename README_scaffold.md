# Physical — Phase 1 scaffold (thin vertical slice)

A tracer bullet through the whole stack: **log a lift → the validated engine
ranks it against the young-male population → the screen shows the tier**, with
bodyweight snapshotted at the time of each lift.

## Stand it up

```bash
flutter create physical            # generate the platform shells (ios/android/…)
cd physical
# add the dependency
flutter pub add flutter_riverpod
flutter pub add dev:test
```

Then drop these files into this layout (replacing the generated lib/main.dart):

```
physical/
├── pubspec.yaml                 # (use the provided one, or just `flutter pub add` as above)
├── lib/
│   ├── main.dart
│   ├── engine/
│   │   └── rank_engine.dart     # the validated engine (pure Dart)
│   ├── data/
│   │   ├── metrics.dart         # metric registry (ranked/tracked/background)
│   │   └── repository.dart      # storage seam (in-memory for now)
│   ├── state/
│   │   └── providers.dart       # Riverpod wiring
│   └── ui/
│       └── home_screen.dart     # the thin-slice UI
└── test/
    ├── rank_engine_test.dart    # parity test vs the Python oracle
    └── golden_vectors.json      # generated by gen_golden.py
```

Note the imports assume that layout (`engine/`, `data/`, `state/`, `ui/`). If
you keep everything flat, adjust the relative import paths.

## Run

```bash
flutter test          # should pass the engine parity suite (51 cases)
flutter run           # launches with a little demo data already ranked
```

Tap **Log** to add a lift. For strength lifts the bodyweight field pre-fills
from your latest bodyweight and is snapshotted onto that lift forever.

## What this is / isn't

- **Is:** the real engine, the three-tier data model, the storage seam, Riverpod
  state, and a working log→rank loop you can feel.
- **Isn't (yet):** the body graph (next increment — ports the prototype's SVG
  figures + badges), persistence (the in-memory repo swaps for local + cloud
  behind the same interface), the graphs page, habits, AI coach.

## Honesty note

The engine's math is validated cross-language (the Dart primitives reproduce the
Python oracle to ~1e-5). The engine + parity test were verified; the UI/state
files were written carefully but not compiled in this environment, so expect a
nit or two on first `flutter run` — we fix those together. Strength standards are
provisional (see STANDARDS_METHODOLOGY.md) and tuning them is a data edit that
flows through automatically.
